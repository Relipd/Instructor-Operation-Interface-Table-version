## 二、核心设计

### 2.1 架构原则

- **一张表闭环**：所有数据（异常记录 + 反馈回传）均读写同一张飞书表，不依赖配置表/副表
- **平台名/部门/指导员**：全部直接从主表对应字段读取，无需跨表映射
- **分级状态**：状态采用「层级一 + 层级二」双级模型

### 2.2 状态层级模型

| 层级 | 存储位置 | 值域 | 作用 |
|---|---|---|---|
| 层级一 | 前端归一化显示 | 待处理 / 整改中 / 已完结 | 前端状态卡、表格徽章、筛选 |
| 层级二 | 底表 `statusFieldId` | 待处理 / 确存风险 / 排除风险 / 整改完结 | 底表流转，精确区分处理路径 |

层级二 → 层级一映射（`parseStatus`）：

| 层级二（底表） | 层级一（前端） |
|---|---|
| 待处理 | 待处理 |
| 确存风险 | 整改中 |
| 排除风险 | 已完结 |
| 整改完结 | 已完结 |

### 2.3 异常类型 → 衍生属性

异常类型通过硬编码映射表推导**风险等级**和**处理时效**：

```
异常类型 → 风险等级（R1/R2/R3）+ 处理时效（- / 1个工作日 / 3个工作日）
```

路径：`src/App.tsx` → `EXCEPTION_TYPE_RISK_MAP` / `EXCEPTION_TYPE_DEADLINE_MAP`

### 2.4 超时判定

`calcTimeout(reviewTime, deadlineText, status)` 根据审阅时间 + 处理时效计算是否超时。已完结状态永不算超时。超时结果回传底表 `feedbackIsTimeoutFieldId`（写「是」）。

## 三、完整业务流程

### 3.1 状态流转图

```
                    待处理
                      │
                 [确认核查]
                      │
          ┌───────────┴───────────┐
          │                       │
     ┌ 排除风险 ┐           ┌ 确存风险 ┐
     │ 填排除理由 │           │ 风险根因 │
     │ 选指导员   │           │ 整改措施 │
     │            │           │ 预计时间 │
     └──┬─────────┘           │ 选指导员   │
        │                     └──┬─────┘
     [确认排除风险]           [确存风险]
        │                       │
        ▼                       ▼
      已完结                  整改中
    (层级二:排除风险)      (层级二:确存风险)
                                │
                           [整改反馈]
                          填写整改情况反馈
                          选择指导员
                                │
                           [确认整改完结]
                                │
                                ▼
                              已完结
                          (层级二:整改完结)
```

### 3.2 第一步：确认核查（ConfirmActionDialog）

**入口**：待处理记录的行操作「确认核查」按钮

**流程**：
1. 选择指导员（部门匹配 > 模糊匹配 > 全量 fallback）
2. 选择风险处置方式 → 显示对应分支表单
3. 提交 → `handleConfirmAction`

**排除风险分支**：
- 写底表：
  - `statusFieldId` = `'排除风险'`（层级二）
  - `feedbackResultFieldId` = 排除理由
  - `feedbackStatusFieldId` = `'排除风险'`
  - `handlerFieldId` = 指导员
- 前端状态：→ `已完结`

**确存风险分支**：
- 写底表：
  - `statusFieldId` = `'确存风险'`（层级二）
  - `feedbackResultFieldId` = `风险根因：xxx\n整改措施：xxx`（合并文本）
  - `expectedCompletionTimeFieldId` = 毫秒时间戳（日期字段特殊处理）
  - `feedbackStatusFieldId` = `'确存风险'`
  - `handlerFieldId` = 指导员
- 前端状态：→ `整改中`

### 3.3 第二步：整改反馈（FeedbackDialog）

**入口**：整改中记录的行操作「整改反馈」按钮

**流程**：
1. 弹窗展示区显示：审阅异常编码、异常类型、风险根因、整改措施、预计整改完成时间
2. 使用预设模板填写整改情况反馈
3. 选择指导员
4. 提交 → `handleFeedbackSubmit`

**写底表**：
- `statusFieldId` = `'整改完结'`（层级二）
- `rectificationFeedbackFieldId` = 整改情况反馈内容
- `feedbackStatusFieldId` = `'整改完结'`
- `handlerFieldId` = 指导员
- 前端状态：→ `已完结`

## 四、技术实现要点

### 4.1 数据读取（fetchData）

- 渐进式拉取：`getRecordsByPage({ pageSize: 200 })`，连续读取无间隔
- pageSize 上限 200（SDK 限制，超过会白屏），100 页封顶（2 万行）
- 渲染节流：首屏立即刷新，后续每 800ms 刷新一次（3k 行约 3-4 次重新渲染）
- 骨架屏：`loading && allRecords.length === 0` 时显示占位动画

### 4.2 数据变更监听（onDataChange）

- 使用 `useRef` 持有最新 config 和 fetchData，避免因依赖变化反复重订阅
- 800ms 防抖合并多次变更
- `timeoutWriteRef` 防止超时回传自身触发的循环触发

### 4.3 指导员映射（instructorMapRef）

- 从主表记录构建：`department → Set<指导员>`（合并业务渠道指导员 + 支持部门指导员，逗号分隔）
- 查找策略：精确匹配 → 模糊匹配（互相包含）→ fallback 全量指导员
- 直接在 `fetchData` 内联构建，无需额外读取副表

### 4.4 回传机制

- 单字段写入：`updateRecordField(tableId, recordId, fieldId, value)` → `table.setCellValue`
- 多字段批量：`updateRecordFields(tableId, recordId, fields)` → `table.setRecord`
- **日期字段特殊处理**：飞书日期字段不接受字符串，必须传毫秒时间戳
- 超时回传：后台异步，并发 5 个，3 秒后清除防循环标记

### 4.5 日期读取（cellToDate / cellToText）

- `cellToDate`：处理数字时间戳、字符串、`{value}` 对象，返回格式化日期字符串
- `cellToText`：处理 `null`/`undefined`/字符串/数字/布尔/数组/`{text}`/`{name}` 等飞书单元格类型

### 4.6 性能优化要点

| 项 | 措施 |
|---|---|
| 3k 行渲染 | 800ms 节流刷新，首屏优先 |
| 指导员映射 | Set 去重 + Ref 避免重构建 |
| 部门列表缓存 | `deptCacheRef`，`tableId + fieldId` 为 key |
| onDataChange 循环 | `timeoutWriteRef` + 3s 窗口 |
| 表格分页 | 每页 20 条，减少 DOM 压力 |
| 组件 memo | ExceptionTable `React.memo` |

## 五、筛选与搜索

### 5.1 前端筛选栏

| 筛选项 | 数据来源 | 默认值 |
|---|---|---|
| 状态 | 固定列表：待处理/整改中/已完结/已超时 | 全部状态 |
| 风险等级 | 动态提取：R3/R2/R1 | 全部等级 |
| 来源平台 | 动态提取：sourcePlatform（中文名） | 全部平台 |
| 异常类型 | 动态提取：exceptionType（中文排序） | 全部类型 |

### 5.2 搜索

支持三个字段的模糊搜索：
- **异常编号** `exceptionCode`
- **业务渠道指导员** `deptInstructors`
- **申请人** `permApplicant || extAccountName`

### 5.3 后台筛选（配置面板）

- **处理人筛选**：按 handler 字段模糊匹配（保存到 config，持久化）
- **部门权限筛选**：checkbox 多选，实时从主表 `departmentFieldId` 提取去重，按部门精确匹配

## 六、组件说明

### 6.1 StatsCards（统计卡片）

- 顶部四列：待处理 / 整改中 / 已完结 / 已超时
- 数量格式：`text-xl font-semibold` + 对应状态色
- 点击筛选联动：再次点击取消筛选
- 活跃筛选态：`ring-2 ring-[#3370ff]`

### 6.2 ExceptionTable（主表格）

- 列：平台 / 异常类型 / 异常详细描述 / 申请人 / 状态 / 审阅时间 / 处理时效 / 操作
- **异常类型、异常详细描述**：使用 Radix Tooltip 悬浮显示全文（Portal 渲染，不受表格 overflow 裁剪）
- 操作按钮：
  - 待处理 → 「确认核查」（蓝色）
  - 整改中 → 「整改反馈」（橙色）
  - 已完结 → 无操作
- 分页：每页 20 条，首/末页、上/下页、跳转
- 导出：Excel 导出全部筛选结果

### 6.3 ConfirmActionDialog（确认核查弹窗）

- 三段式布局：Header 固定 + 滚动区 + 关闭
- 选择页：排除风险（淡绿文字）/ 确存风险（暗红文字）
- 排除风险分支：理由 + 「确认排除风险」（绿色按钮）
- 确存风险分支：根因 + 措施 + 预计时间 + 「确存风险」（橙色按钮）
- 指导员下拉：部门匹配，空时提示

### 6.4 FeedbackDialog（整改反馈弹窗）

- 三段式布局：Header 固定 + 展示区/表单滚动 + Footer 固定
- 展示区：审阅异常编码 / 异常类型 / 风险根因 / 整改措施 / 预计时间（纵向排列）
- 解析「风险根因：xxx\n整改措施：xxx」格式的合并文本
- 整改情况反馈预设模板：整改进度、完成情况、异常处理说明、影响范围、验证结果、预防措施

### 6.5 DetailDrawer（详情抽屉）

- Sheet 侧边栏（640px），纵向滚动
- 分区：基本信息 / 异常描述 / 反馈内容 / 权限中心 / 外部系统
- FieldRow 组件：空值自动隐藏

### 6.6 ConfigPanel（配置面板）

- 数据源：多维表格选择 + 数据表选择（按名 A→Z 排序）
- 字段映射：四组（基本信息 / 权限中心 / 外部系统 / 反馈字段），每组独立显示
- 部门权限筛选：checkbox 列表，实时从主表读取，带缓存（`deptCacheRef`）、防并发（`deptLoadingRef`）
- 字段加载：`getActiveView()` 带 5s 超时，失败则回退 `getFieldMetaList()`

## 七、类型定义速查

### 7.1 ExceptionRecord

```typescript
ExceptionRecord {
  // 基础
  id, recordId, exceptionCode         // 标识
  exceptionType, sourcePlatform       // 分类
  riskLevel, status                   // 状态
  handler, department                 // 归属
  reviewTime, processDeadline, deadlineText  // 时效
  isTimeout, deptInstructors          // 计算字段
  
  // 整改
  expectedCompletionTime           // 预计整改完成时间
  rectificationFeedback            // 整改情况反馈
  riskAction                       // 风险处置（排除风险/确存风险/整改完结）
  
  // 权限中心
  permAccountName, permStoreName, permWorkOrderId,
  permApplicant, permApplicantEmail, permApplicantPhone,
  permUserNickname, permUserRoleList
  
  // 外部系统
  extStoreName, extUserPhone, extAccountName,
  extUserName, extUserRoleList
}
```

### 7.2 IPluginConfig（配置面板绑定字段）

- **基础字段**：`exceptionCodeFieldId`, `platformFieldId`, `dataPlatformNameFieldId`, `statusFieldId`, `exceptionTypeFieldId`, `exceptionDetailFieldId`, `reviewTimeFieldId`, `departmentFieldId`, `deptInstructorsFieldId`, `sharedInstructorsFieldId`, `handlerFieldId`
- **权限中心字段**：`permAccountNameFieldId` ~ `permUserRoleListFieldId`（8 个）
- **外部系统字段**：`extStoreNameFieldId` ~ `extUserRoleListFieldId`（5 个）
- **反馈字段**：`feedbackRecordCodeFieldId`, `feedbackStatusFieldId`, `feedbackResultFieldId`, `feedbackTimeFieldId`, `feedbackPersonFieldId`, `feedbackIsTimeoutFieldId`, `expectedCompletionTimeFieldId`, `rectificationFeedbackFieldId`
- **筛选**：`handlerNameFilter`, `departmentNameFilter`

## 八、运行与调试

### 开发环境

```bash
cd 项目根目录
npm install
npx vite --port 5179
```

飞书插件开发模式下访问 `http://localhost:5179/`。

### 类型检查与构建

```bash
npx tsc --noEmit          # TypeScript 类型检查（零错误）
npx vite build             # 生产构建
```

### 注意事项

1. **飞书 SDK 限制**：`getRecordsByPage` 单页上限 200 条，超出会白屏
2. **日期字段回传**：必须传毫秒时间戳，不能传字符串
3. **HMR 闭包陈旧**：修改核心逻辑后建议硬刷新（Ctrl+Shift+R）而非依赖 HMR
4. **onDataChange 循环**：写操作会触发 onDataChange，通过 `timeoutWriteRef` + 3s 窗口防范
5. **开发服务器端口**：默认 5179，如被占用会自动尝试下一端口
